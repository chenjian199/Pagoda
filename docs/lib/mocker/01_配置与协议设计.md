# Pagoda Mocker 配置与协议设计

## 1. 模块目标

本文件定义 Pagoda Mocker 的公共配置、请求协议、输出协议和内部事件协议。

该部分继承原模块中 `common/protocols.rs` 的核心设计，但按 Pagoda 当前范围做裁剪：

```text
保留：
  MockEngineArgs 核心配置
  DirectRequest
  OutputSignal
  WorkerType
  PreemptionMode
  ReasoningConfig
  PerfModel 引用
  KvEventPublishers
  FpmPublisher
  ForwardPassSnapshot
  MoveBlock 语义

移除：
  EngineType::Sglang
  SglangArgs
  KVBM G2 offload 字段
  KVBM G1/G2 语义暴露

新增：
  LMCache mock adapter 配置
  External shared cache 模拟开关
  LMCache hit/save latency 模拟参数
```

---

## 2. 引擎类型

Pagoda 第一阶段只支持 vLLM 引擎模拟。

因此不再需要：

```rust
enum EngineType {
    Vllm,
    Sglang,
}
```

推荐简化为：

```rust
enum MockBackendKind {
    Vllm,
}
```

或者直接移除 engine type 字段，由 `create_engine()` 固定创建 vLLM scheduler。

如果未来要接其它后端，重新加入：

```rust
enum MockBackendKind {
    Vllm,
    Sglang,
    Custom(String),
}
```

但第一阶段不实现。

## 3. Worker 类型

保留原 `WorkerType` 设计：

```rust
enum WorkerType {
    Aggregated,
    Prefill,
    Decode,
}
```

语义：

```text
Aggregated:
  单 worker 同时模拟 prefill 和 decode。

Prefill:
  P/D 分离中的 prefill worker；
  负责模拟 prompt KV 生成；
  首个 decode token 可视为 prefill pass 的输出；
  可产生 handoff_delay_ms。

Decode:
  P/D 分离中的 decode worker；
  不计算 prefill；
  接收已完成 prefill / KV handoff 后继续 decode。
```

第一阶段优先使用 `Aggregated`。  
`Prefill` / `Decode` 保留给 vLLM disaggregated replay 与 LMCache handoff 模拟。

## 4. 请求协议：DirectRequest

保留原结构：

```rust
struct DirectRequest {
    tokens: Vec<Token>,
    max_output_tokens: usize,
    uuid: Option<Uuid>,
    dp_rank: u32,
    arrival_timestamp_ms: Option<f64>,
}
```

字段说明：

```text
tokens:
  已经分词后的 token id。
  Mocker 不负责 tokenizer。

max_output_tokens:
  最多模拟生成多少个输出 token。

uuid:
  可选请求 ID。
  如果为空，mocker 生成新 UUID。

dp_rank:
  数据并行 rank。
  用于模拟多 worker / 多 rank 场景。

arrival_timestamp_ms:
  请求到达时间。
  replay 中用于统计真实到达时间与排队延迟。
```

约束：

```text
DirectRequest 不包含原始 prompt 文本；
DirectRequest 不包含模型参数；
DirectRequest 不包含 HTTP 协议字段；
DirectRequest 不包含真实 KV tensor。
```

## 5. 输出协议：OutputSignal

保留原结构：

```rust
struct OutputSignal {
    uuid: Uuid,
    completed: bool,
    handoff_delay_ms: Option<f64>,
}
```

字段说明：

```text
uuid:
  对应请求 ID。

completed:
  当前 token 输出后，请求是否已完成。

handoff_delay_ms:
  P/D 分离中 prefill worker 到 decode worker 的模拟 KV handoff 延迟。
```

`handoff_delay_ms` 计算依赖：

```text
input_tokens
kv_bytes_per_token
kv_transfer_bandwidth
worker_type
completed
```

Aggregated 模式下通常为空。

## 6. Mocker 主配置：PagodaMockEngineArgs

建议保留原 `MockEngineArgs` 的核心字段，并重命名为：

```rust
struct PagodaMockEngineArgs
```

### 6.1 KV 容量与 block 配置

```rust
num_gpu_blocks: usize
block_size: usize
```

语义：

```text
num_gpu_blocks:
  模拟 vLLM GPU paged KV buffer 中有多少个 block。

block_size:
  每个 KV block 覆盖多少 token。
```

默认值建议：

```text
block_size = 64
num_gpu_blocks = 16384
```

### 6.2 调度容量配置

```rust
max_num_seqs: Option<usize>
max_num_batched_tokens: Option<usize>
```

语义：

```text
max_num_seqs:
  最多同时 running 的请求数。

max_num_batched_tokens:
  单个 scheduler pass 中最多处理多少 token。
```

### 6.3 Prefix caching 与 chunked prefill

```rust
enable_prefix_caching: bool
enable_chunked_prefill: bool
```

语义：

```text
enable_prefix_caching:
  是否允许相同 prefix 复用。

enable_chunked_prefill:
  长 prompt 是否可以按 token budget 分多个 pass prefill。
```

### 6.4 模拟速度

```rust
speedup_ratio: f64
decode_speedup_ratio: f64
```

语义：

```text
speedup_ratio:
  同时作用于 prefill 和 decode。
  >1 表示模拟更快。

decode_speedup_ratio:
  只作用于 decode。
  可模拟 speculative decoding 或 decode 加速。
```

### 6.5 并行与启动配置

```rust
dp_size: u32
startup_time: Option<f64>
worker_type: WorkerType
```

语义：

```text
dp_size:
  数据并行 worker 数。

startup_time:
  模拟 worker 初始化耗时。

worker_type:
  Aggregated / Prefill / Decode。
```

### 6.6 性能模型配置

```rust
planner_profile_data: Option<PathBuf>
perf_model: Arc<PerfModel>
aic_backend: Option<String>
aic_system: Option<String>
aic_backend_version: Option<String>
aic_tp_size: Option<usize>
aic_model_path: Option<String>
aic_moe_tp_size: Option<usize>
aic_moe_ep_size: Option<usize>
aic_attention_dp_size: Option<usize>
```

第一阶段保留这些字段，但要求：

```text
aic_backend 只接受 "vllm" 或 Pagoda 自己定义的 vLLM backend 名；
SGLang backend 不作为有效值。
```

### 6.7 Router 与本地 indexer 配置

```rust
enable_local_indexer: bool
router_queue_policy: Option<RouterQueuePolicy>
```

用于 replay / kv_router 模式。

### 6.8 P/D 分离传输配置

```rust
bootstrap_port: Option<u16>
kv_bytes_per_token: Option<usize>
kv_transfer_bandwidth: Option<f64>
```

语义：

```text
bootstrap_port:
  prefill/decode rendezvous 端口。
  第一阶段可保留，具体网络 bootstrap 可简化。

kv_bytes_per_token:
  每个 token 的 KV 字节数，用于估算 handoff delay。

kv_transfer_bandwidth:
  KV 传输带宽，单位 GB/s。
```

### 6.9 Reasoning token 模拟

```rust
reasoning: Option<ReasoningConfig>
```

用于模拟模型输出中带 thinking token 的场景。

### 6.10 KV event / ZMQ 配置

```rust
zmq_kv_events_port: Option<u16>
zmq_replay_port: Option<u16>
```

语义：

```text
zmq_kv_events_port:
  模拟 vLLM 原生 ZMQ KV event publisher。

zmq_replay_port:
  用于回放 buffered KV event batch。
```

### 6.11 Preemption 配置

```rust
preemption_mode: PreemptionMode
```

保留：

```rust
enum PreemptionMode {
    Lifo,
    Fifo,
}
```

默认 `Lifo`，模拟 vLLM v1 默认更接近的抢占策略。

## 7. 删除的配置

以下字段不进入 Pagoda 第一阶段公开配置：

```rust
num_g2_blocks
offload_batch_size
bandwidth_g1_to_g2_gbps
bandwidth_g2_to_g1_gbps
sglang
```

原因：

```text
num_g2_blocks / offload_batch_size / bandwidth_g1_to_g2_gbps / bandwidth_g2_to_g1_gbps:
  属于 KVBM G1↔G2 offload 模拟。
  Pagoda 第一阶段不做 KVBM / memory。

sglang:
  属于 SGLang scheduler。
  Pagoda 第一阶段只做 vLLM 引擎版本。
```

## 8. 新增 LMCache 模拟配置

新增：

```rust
struct LmCacheMockConfig {
    enabled: bool,
    enable_l1: bool,
    enable_l2: bool,
    l1_capacity_blocks: Option<usize>,
    l2_capacity_blocks: Option<usize>,
    l1_hit_latency_ms: f64,
    l2_hit_latency_ms: f64,
    save_latency_ms: f64,
    publish_external_events: bool,
    shared_metadata_lookup: bool,
}
```

字段说明：

```text
enabled:
  是否启用 LMCache 外部缓存模拟。

enable_l1:
  是否模拟 LMCache L1 内存缓存。

enable_l2:
  是否模拟 LMCache L2 持久化后端。

l1_capacity_blocks / l2_capacity_blocks:
  可选容量限制。
  第一阶段可不实现复杂 eviction，只保留容量接口。

l1_hit_latency_ms:
  L1 命中加载延迟。

l2_hit_latency_ms:
  L2 命中加载延迟。

save_latency_ms:
  保存新 block 到 LMCache 的模拟延迟。

publish_external_events:
  是否对 LMCache external shared tier 发布 Stored / Removed 事件。

shared_metadata_lookup:
  是否允许 router 通过 shared metadata query 查询 LMCache 是否存在 block。
```

第一阶段可先实现：

```text
enabled
l1_hit_latency_ms
l2_hit_latency_ms
save_latency_ms
publish_external_events
shared_metadata_lookup
```

容量和 eviction 可以后续补齐。

## 9. KV event publisher 协议

保留原 `KvEventPublishers` 思路：

```rust
struct KvEventPublishers {
    event_sink: Option<Arc<dyn KvCacheEventSink>>,
    raw_sink: Option<Arc<dyn RawKvEventSink>>,
}
```

### 9.1 标准事件 sink

```rust
trait KvCacheEventSink {
    fn publish(&self, event: KvCacheEvent) -> anyhow::Result<()>;

    fn publish_with_storage_tier(
        &self,
        event: KvCacheEvent,
        storage_tier: StorageTier,
    ) -> anyhow::Result<()>;
}
```

用于 RouterEvent / event plane。

### 9.2 Raw 事件 sink

```rust
trait RawKvEventSink {
    fn publish(&self, event: RawKvEvent) -> anyhow::Result<()>;
}
```

用于模拟 vLLM native ZMQ event。

### 9.3 RawKvEvent

```rust
struct RawKvEvent {
    event: KvCacheEvent,
    block_token_ids: Option<Vec<Vec<u32>>>,
    storage_tier: StorageTier,
}
```

保留原因：

```text
标准 RouterEvent 只需要 block hash；
ZMQ / vLLM raw event 可能需要 block token ids；
storage_tier 用于区分 Device / ExternalShared 等层级。
```

## 11. MoveBlock 语义保留但去 KVBM 化

保留三类行为：

```rust
enum MoveBlock {
    Use(...),
    Deref(...),
    Promote(...),
}
```

但在 Pagoda 文档中解释为：

```text
Use:
  请求需要使用一组 KV block。
  Local cache / LMCache 命中可复用，未命中则分配本地 vLLM KV block。

Deref:
  请求释放持有的 block reference。
  block 可转入 inactive prefix cache 或被淘汰。

Promote:
  partial block 填满后晋升为 full block。
  full block 可进入 prefix cache，并可保存到 LMCache。
```

不再解释为：

```text
MutableBlock / ImmutableBlock / kvbm-logical RAII 生命周期
```

---

## 12. 设计约束

```text
配置必须可序列化；
配置必须可从 JSON extra args 加载；
不支持的字段必须显式报错；
删除字段不能静默接受；
LMCache 模拟字段必须与真实 LMCache connector 字段解耦；
Mocker 不直接启动真实 LMCache server；
Mocker 不真实读写 KV tensor。
```
